================================================================================
KEY TECHNICAL FINDINGS
================================================================================

1. PRIMARY API ENDPOINT
   - URL: GET https://claude.ai/api/organizations/{orgId}/usage
   - Authentication: HTTP cookies (lastActiveOrg cookie contains orgId)
   - Response: JSON with utilization percentages and reset times
   - No special API keys required

2. RESPONSE DATA STRUCTURE
   {
     "five_hour": {
       "utilization": <0-100>,
       "resets_at": "<ISO8601 timestamp>"
     },
     "seven_day": { ... },
     "seven_day_sonnet": { ... },
     "seven_day_opus": { ... },
     "extra_usage": {
       "is_enabled": <boolean>,
       "monthly_limit": <cents>,
       "used_credits": <cents>
     }
   }

3. RELATED ENDPOINTS
   - /api/bootstrap/{orgId}/app_start → Subscription tier detection
   - /api/organizations/{orgId}/prepaid/credits → Credit balance
   - /api/organizations/{orgId}/chat_conversations/{id} → Token counting
   - /api/organizations/{orgId}/memory → Memory data

4. AUTHENTICATION MECHANISM
   - Method: Browser cookie-based (HTTP-only)
   - Key Cookie: "lastActiveOrg" = organization UUID
   - Flow: Automatic inclusion with fetch requests
   - Multi-account: Support via Firefox container headers

5. SUBSCRIPTION TIERS DETECTED
   - claude_free: Free tier
   - claude_pro: Pro subscription
   - claude_team: Team/business accounts
   - claude_max_5x: Max tier with 5x multiplier
   - claude_max_20x: Max tier with 20x multiplier

================================================================================
RECOMMENDED IMPLEMENTATION APPROACH
================================================================================

ARCHITECTURE (3 Components):
  1. Background Service Worker
     - Manages all API calls to Claude.ai
     - Stores usage data in browser.storage.local
     - Implements polling and error handling
     - Rate limit detection and backoff

  2. Content Script
     - Runs on claude.ai website
     - Extracts organization ID from cookies or DOM
     - Injects UI overlays for usage display
     - Communicates with background service worker

  3. API Wrapper (ClaudeAPI class)
     - Encapsulates all API interactions
     - Handles authentication and CORS
     - Manages subscription tier caching
     - Implements retry logic

POLLING STRATEGY:
  - Background: Every 5 minutes (default)
  - Message Events: Immediately after API calls complete
  - Peak Hours: Every 2-3 minutes (1pm-7pm GMT weekdays)
  - Rate Limited: Exponential backoff (1s, 2s, 4s, etc.)

DATA FLOW:
  1. User opens claude.ai
  2. Content script extracts organization ID
  3. Background worker fetches usage data via API
  4. Data stored in chrome.storage.local
  5. UI updated on all tabs
  6. Polling continues at regular intervals
  7. On message send: capture pre-send usage
  8. On message complete: capture post-send and calculate delta

================================================================================
CRITICAL IMPLEMENTATION DETAILS
================================================================================

1. ORGANIZATION ID EXTRACTION
   Priority 1: lastActiveOrg cookie
   Priority 2: Content script message handler
   Priority 3: URL pattern matching in DOM

2. ERROR HANDLING
   - 429 (Rate Limited): Exponential backoff retry
   - 401 (Unauthorized): Cookie expired, redirect user
   - 500 (Server Error): Retry after delay
   - Network Error: Use cached data with "stale" indicator

3. MULTI-ACCOUNT SUPPORT
   - Use Firefox container IDs (cookieStoreId)
   - Store data separately per container
   - Add X-Container header to requests

4. PERFORMANCE OPTIMIZATION
   - Cache subscription tier (24-hour TTL)
   - Cache usage data (2-minute TTL)
   - Batch API calls where possible
   - Implement request deduplication

5. SECURITY BEST PRACTICES
   - Never expose org IDs in logs
   - Validate IDs before API calls
   - Use credentials: 'include' for auth
   - Store in browser.storage.local (encrypted)

================================================================================
TESTING & TROUBLESHOOTING
================================================================================

Verification Steps:
  1. Load extension in chrome://extensions
  2. Open claude.ai in tab
  3. Check extension popup - should show usage percentages
  4. Verify data in DevTools → Application → Storage → Local Storage

Common Issues:
  - No data: Check that you're logged into claude.ai
  - 429 errors: Rate limiting, implement backoff
  - Wrong org ID: Try logging out/in of claude.ai
  - Stale data: Check polling interval configuration

CONCLUSION
================================================================================

Claude.ai exposes a well-documented internal API that provides real-time usage
quota information. Chrome extensions can access this API using standard browser
authentication (cookies), making it straightforward to build quota tracking tools.

The reference implementation demonstrates this is a proven, stable approach that
can be productionized. With proper error handling and caching, extensions can
reliably display usage information to users.

Key Success Factors:
  1. Cookie-based authentication (no complex auth needed)
  2. Well-structured API response format
  3. Reliable polling mechanism
  4. Proper error handling and backoff
  5. Storage and caching strategy

This research provides everything needed to implement a production-grade
Claude quota tracking Chrome extension.
